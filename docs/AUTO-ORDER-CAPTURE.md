# Авто-фиксация заказов и сумм — без ввода (план)

**Статус: PLANNED (не реализовано).** Владелец зафиксировал проблему: отмечать
«Подача/Везу/Высадил» руками (или голосом) — двойная работа, ведь заказ он уже
принимает в приложении Bolt. AION должен узнавать **сам**: что пришёл заказ
(например, на 4 €), что он выполнен, — и **сам записывать доход**, без ручного
ввода сумм.

## Зачем это вообще

Итог смены (все км — включая холостые до точек, топливо по чекам, доход,
прибыль) считается **без всяких отметок** — это база, она уже работает.
Отметки нужны только для деления км на классы (заказ/подача/порожняк) и
распределения топлива по ним. Ценность реальная, но она не должна стоить
водителю ни одного нажатия.

## Реальный поток заказа в Bolt (со слов владельца)

1. Приходит заказ → на экране карточка: **куда** ехать, сумма, **наличные/карта**.
2. Водитель принимает.
3. Bolt **сам открывает карту с маршрутом до точки**.

Точка назначения, таким образом, доступна двумя путями:
- **Навигатор-интент (этап 1, УЖЕ сделано):** когда Bolt даёт «Навигация», Android
  предлагает выбор навигатора → водитель выбирает AION → AION открывается с точкой
  и маршрутом по дорогам (OSRM). Требует одного тапа «Навигация» + выбора AION.
- **Чтение экрана (Accessibility, автомат):** AION сам считывает адрес/сумму/оплату
  с карточки заказа и с открытой картой Bolt — без тапов. Требует нативного модуля
  + разрешения пользователя. Это и есть «полный автомат».

## Где данные (уточнено владельцем)

Полная карточка заказа — **на ЭКРАНЕ** Bolt: куда ехать, сумма, и **способ
оплаты (наличные / карта)**. Уведомления могут не содержать всего этого →
основной источник, скорее всего, **Accessibility Service** (чтение экрана Bolt
с явного разрешения пользователя), а не уведомления. Парсер один и тот же:
текст (с экрана ИЛИ из уведомления) → событие заказа.

## Что именно фиксируем автоматически

1. **Фазы заказа** (назначен → еду за пассажиром → везу → завершён) →
   `OrderActivityWindow` для классов км.
2. **Суммы заказов** (заказ на 4 € выполнен → +4 € в доход смены) →
   income ledger, без ручного/голосового ввода.
3. **Способ оплаты — наличные vs карта** (важно: наличка сразу в кармане, карта
   приходит позже; для реального дохода это разное). Уже в модели:
   `incomeDraft.paymentMethod: "cash" | "card" | null` (null = Bolt не показал,
   не выдумываем). Парсер: «наличными/нал» → cash, «картой/безнал» → card.
4. **Адрес назначения** («куда») — из карточки; для показа точки на карте
   (этап навигатора). Извлечение адреса — следующий шаг.

## Варианты источника (по честности и трудозатратам)

1. **Уведомления Bolt (Android NotificationListenerService)** — рекомендуемый
   для фаз; возможно, и для сумм. Bolt показывает уведомления при назначении
   заказа, начале и завершении поездки. Слушатель уведомлений (с явного
   разрешения пользователя) переводит их в `OrderActivityWindow` с
   `evidence: "platform_import"` (контракт уже предусматривает). Если в текстах
   уведомлений есть цена заказа — оттуда же берём сумму в доход.
   - − Нативный модуль (Expo config plugin / кастомный dev client) → нужна
     пересборка APK (EAS) и решение владельца.
   - − Формат уведомлений Bolt не документирован — нужно снять реальные тексты
     уведомлений с устройства (RU/ET локали) до реализации. Есть ли в них
     суммы — узнаем только по реальным скриншотам шторки.
   - + Ноль нажатий; работает в фоне; честные события платформы.

2. **Accessibility Service (чтение экрана Bolt)** — если сумм в уведомлениях
   нет. Сервис специальных возможностей (с явного включения пользователем в
   настройках Android) может читать карточку заказа на экране Bolt — там цена
   есть всегда. Самый полный источник, но самый инвазивный и хрупкий к
   обновлениям Bolt; тоже нативный модуль. Рассматривать вторым шагом, только
   если уведомления не содержат сумм.

3. **Импорт скриншотов Bolt (OCR — УЖЕ РАБОТАЕТ) — полуавтомат на сегодня.**
   Один скриншот заработка Bolt после смены → AION распознаёт все поездки с
   суммами и записывает доход. Это уже закрывает «сам вбивать суммы» ценой
   одного скриншота. Для окон км скриншот не годится: в `OcrTripRow` только
   `occurredAt` (одна временная точка), начала/конца поездки нет — длительность
   выдумывать нельзя.

4. **Ручные отметки / голос** — остаются как необязательный запасной способ.
   UI уже помечен «необязательно»; итоги смены от них не зависят.

## Решение

- Классы км — **опциональная надстройка**; база (общие км/топливо/прибыль)
  никогда не требует отметок.
- Автозахват строим на уведомлениях Bolt (вариант 1) после: (а) снятия реальных
  текстов уведомлений с устройства владельца — по ним же решаем, есть ли там
  суммы; (б) решения владельца на нативную пересборку APK. Если сумм в
  уведомлениях нет — оцениваем Accessibility-путь (вариант 2) отдельным
  решением владельца.
- До этого «сам вбивает суммы» уже доступен полуавтоматом: скриншот заработка
  Bolt после смены → OCR-импорт всех заказов (вариант 3).
- Никакой классификации «по догадке» (остановился → значит посадил) — это
  выдумывание данных, запрещено принципом проекта. Суммы — только из реальных
  источников (уведомление/экран/скриншот), никогда «средним чеком».

## Что уже подготовлено (без устройства) — JS-конвейер ГОТОВ и покрыт тестами

- `features/orders/boltNotificationCapture.ts`: `parseBoltNotification` (уведомление →
  событие заказа), `applyCapturedOrderEvent` (событие → окна км + черновик дохода).
  Извлечение суммы устойчиво к формату: символ до/после числа (`€4.50`, `EUR 5.20`,
  `4,50 €`), не путает расстояние («4.5 km») с деньгами; оплата RU/EN/ET
  (наличные/картой, cash/card, sularaha/kaart). Тест `test-bolt-notification-capture.mjs`
  — 10 кейсов.
- `features/orders/orderCaptureRunner.ts`: `processBoltNotifications(shiftId, notifications)`
  — единственная точка входа для нативного слушателя: сортировка по времени,
  сериализация конкурентных пачек, персист окон, анти-дубль дохода. Контракт
  `incomeDrafts` несёт `paymentMethod` (наличные/карта). Тест
  `test-order-capture-runner.mjs` — 6 кейсов.

ЖДЁТ сверки с реальными текстами уведомлений: только таблица паттернов ТИПА
события `INITIAL_PATTERNS` (assigned/started/finished) — точные фразы Bolt. Всё
извлечение суммы/оплаты уже формат-независимо. Нативному модулю останется только
поймать уведомление и вызвать `processBoltNotifications`.

## Нативный слушатель — implementation-ready спека (строить, когда можно проверить)

**НЕ реализовано намеренно:** нативную фоновую службу нельзя писать вслепую —
урок орбиты (крашила при сворачивании). Строим только когда есть возможность
собрать APK и проверить на телефоне. Ниже — точный план, чтобы сделать с первого
раза чисто.

Паттерн — как рабочий `plugins/withAionOverlayOrb.js` (copy .kt → withDangerousMod,
service/permission → withAndroidManifest, регистрация пакета → withMainApplication):

1. `native-modules/aion-notif-capture/AionNotifListenerService.kt` — extends
   `NotificationListenerService`. `onNotificationPosted(sbn)`: **весь код в try/catch**;
   если `sbn.packageName ∈ BOLT_DRIVER_PACKAGES` → достать `extras` (title/text),
   отдать в кольцевой буфер (последние ~50) + эмитнуть событие в JS. Больше НИЧЕГО
   (без overlay, без foreground-сервиса, без потоков, без persistence) — минимальная
   поверхность краша.
2. `AionNotifCaptureModule.kt` (ReactContextBaseJavaModule): `isAccessGranted()`
   (`Settings.Secure.getString(..., "enabled_notification_listeners")` contains pkg),
   `openAccessSettings()` (`ACTION_NOTIFICATION_LISTENER_SETTINGS`), `drainBuffer()`
   (отдать накопленные уведомления), + `DeviceEventEmitter` "aionBoltNotification".
3. `plugins/withAionNotifCapture.js`: манифест — `<service>` с
   `android.permission.BIND_NOTIFICATION_LISTENER_SERVICE` и intent-filter
   `android.service.notification.NotificationListenerService`.
4. JS-мост `features/orders/boltNotificationBridge.ts`: подписка на событие →
   `processBoltNotifications(activeShift.id, [...])` → `incomeDraft` → `addIncome`.
   Полностью тестируем моками (пишем ВМЕСТЕ с нативным, но проверяем в JS).

**Двойной гейтинг безопасности (обязателен — «не проебись»):**
- OS-уровень: слушатель не работает, пока пользователь не включит «Доступ к
  уведомлениям» для AION в настройках Android (по умолчанию ВЫКЛ).
- App-уровень: флаг `deviceSettings.autoOrderCaptureEnabled` (по умолчанию ВЫКЛ) —
  JS игнорирует события, пока владелец явно не включит «Авто-заказы (бета)».
- → у обычного пользователя модуль полностью инертен; активируется только двумя
  осознанными действиями.

**Как проверим на телефоне (даёт реальные тексты уведомлений):** после сборки APK
с модулем — экран «Авто-заказы (бета)» показывает СЫРЫЕ пойманные уведомления
Bolt (buffer). Владелец включает доступ, делает одну поездку → видим точные
тексты assigned/started/finished → финализируем `INITIAL_PATTERNS`. Модуль
сам себя разблокирует.

**Гейт активации:** нативное изменение → bump version/versionCode → новый APK.
Не слать через OTA (нативный код). Пока модуль не вкомпилен — `app.config.ts`
plugins его НЕ содержит.
