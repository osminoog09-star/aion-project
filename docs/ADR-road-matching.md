# ADR / spike — Road Matching (map-matching GPS → дорожный граф)

**Статус: PROPOSED (спайк/проект, НЕ реализовано).** Это design-документ для roadmap-пункта
`maps-road-matching`. Кода/движка нет. Реализацию не начинать без накопленных реальных
треков и явного решения владельца. Никаких выдуманных метрик.

## Контекст

Сейчас у нас есть сырые GPS-точки смены (`GpsTripPointRecord{ t, lat, lng, dM }`,
`ShiftGpsSession`) и дельта-дистанция по точкам. Уже построено:
- классификация км по классам (заказ/подача/порожняк/личное) — `classifyKilometers`;
- распределение топлива по классам — `allocateFuelByKmClass`;
- честные снапшоты — `buildMaps/FuelIntelligenceSnapshot`.

Чего нет: привязки трека к реальной дорожной сети. Дистанция считается по сырым точкам
(haversine между принятыми фиксациями), а не по фактическому проезду по дорогам.

## Проблема

Для следующего этапа (`fuel-maps-predictive`, операционная карта, точная экономика поездки)
нужен **map-matching**: «прилепить» последовательность GPS-точек к рёбрам дорожного графа,
получив реальный маршрут, корректную дистанцию по дорогам и сегменты улиц. Это повышает
точность километража и классов км и открывает предиктивный расход на маршруте.

## Варианты

1. **Внешний API map-matching** (Mapbox Map Matching / Google Roads / HERE).
   - + Быстро, точно, без своей инфраструктуры.
   - − Платно по объёму, vendor lock-in, приватность (треки уходят наружу), лимиты.

2. **Self-host OSRM или Valhalla + OSM-экстракт Эстонии.**
   - + Контроль, без vendor lock-in (соответствует «self-host architecture» из roadmap),
     приватность (треки не уходят), оффлайн-эксперименты на накопленных сменах.
   - − Нужна инфраструктура (сервер, сборка графа из `.osm.pbf` Estonia), devops.

3. **Лёгкий клиентский матчинг к OSM** (через Overpass — в проекте уже есть `overpassFuel.ts`).
   - + Без отдельного сервера, переиспользует существующее.
   - − Overpass не предназначен для map-matching; точность/нагрузка ограничены.

## Рекомендация (спайк)

- **Этап 1 (оффлайн-спайк, без продакшена):** взять накопленные реальные треки смен,
  прогнать через **self-host OSRM** с OSM-экстрактом Эстонии (вариант 2) локально, сравнить
  road-matched дистанцию с текущей haversine-дистанцией. Оценить точность в Пярну.
- **Этап 2:** если точность оправдана — вынести в сервис (или периодический батч на бэке),
  кормить результатом `MapsIntelligenceSnapshot` (реальный маршрут, road-matched км по классам).
- Внешний API (вариант 1) — только как запасной/быстрый старт, с учётом приватности и стоимости.
- Клиентский Overpass-матчинг (вариант 3) — не для продакшена, максимум для прототипа.

## Данные и зависимости

- Вход: реальные `ShiftGpsSession.points` (есть), достаточный объём смен.
- Дорожный граф: OSM-экстракт Эстонии (`.osm.pbf`) для OSRM/Valhalla.
- Зависит от: `route-gps-store` (есть), `maps-gps-intelligence` (классы км — есть).
- Кормит: `fuel-maps-predictive` (предиктивный расход на маршруте).

## Риски

- Точность map-matching на коротких городских поездках Пярну.
- Инфраструктура/стоимость self-host vs приватность/стоимость внешнего API.
- Объём реальных треков: без достаточных данных спайк не репрезентативен.

## Решение

Зафиксировать намерение: **road matching строим как спайк на self-host OSRM + OSM Estonia,
оффлайн, на реальных треках; production-движок — после оценки точности и решения владельца.**
До этого `maps-road-matching` остаётся `blocked`/планом; метрики маршрута не выдумываем.
