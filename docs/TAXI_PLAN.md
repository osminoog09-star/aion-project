# Taxi Plan / «Где таксовать» — planned feature

**Статус: PLANNED (в плане, не реализовано).** Этот документ — только описание будущей
функции для roadmap. Кода/реализации нет; числа и прогнозы не показываются, пока нет
реальных данных (правило roadmap: «absent data stays absent»).

## Суть

Приложение AION Driver показывает водителю Bolt/такси **план на неделю и месяц** — где и
когда выгоднее работать в Пярну: какие районы выбирать, когда ждать высокий спрос, какие
городские события влияют на поездки. Цель — помочь спланировать смены, а не угадывать.

## Что входит (план)

- план на неделю;
- план на месяц;
- горячие зоны;
- мероприятия Пярну;
- прогноз спроса: `low` / `medium` / `high` / `very_high`;
- лучшие часы для работы;
- лучшие зоны ожидания;
- предупреждения о слабом спросе;
- интеграция событий Visit Pärnu;
- weather API (погода);
- public holidays (праздники);
- исторические данные прибыли смен пользователя;
- синхронизация через Supabase;
- еженедельное уведомление «Совет недели для Bolt»;
- уведомления перед крупными событиями.

## Логика прогноза спроса (для будущей реализации)

- пятница вечер/ночь — высокий спрос;
- суббота день/ночь — высокий или очень высокий;
- воскресенье день — средний;
- будни утром и после работы — средний;
- будни ночью — низкий;
- плохая погода может повышать спрос на короткие поездки;
- концерты, фестивали, городские события повышают спрос;
- летом пляж и центр Пярну особенно важны.

Прогноз должен опираться на **реальные** исторические смены водителя + внешние сигналы
(события, погода, праздники), а не на выдуманные значения. Без достаточных данных пункт
показывает «недостаточно данных», а не фейковую оценку.

## Горячие зоны (Пярну)

Pärnu Beach · Rüütli / центр города · автовокзал (Bus Station) · спа-отели (Spa hotels) ·
Nightclub Sunset · концертный зал (Concert Hall) · фестивальные площадки (Festival areas) ·
порт / отели (Port / hotels area).

## Источники данных (план)

- Visit Pärnu events (городские мероприятия);
- weather API (погода);
- public holidays (календарь праздников);
- история смен и прибыли водителя (локально + Supabase);
- горячие зоны и сезонность (пляж/центр летом).

## Связь с существующим

- опирается на накопленные реальные смены (см. `route-gps-store`, post-shift analytics);
- переиспользует классы км и intelligence-слой (`features/intelligence/*`);
- синхронизация и хранение — через Supabase (как у остальных данных смены);
- уведомления — через существующий слой нотификаций приложения.

## Зависимости / гейты

- нужен значимый объём реальных смен (профит/часы/зоны) — иначе план не строится честно;
- внешние API (события/погода/праздники) — отдельная интеграция, ключи и лимиты;
- реализация — после стабилизации Maps GPS Intelligence (классы км) и накопления истории.

> Реализацию НЕ начинать без явного решения владельца и достаточных реальных данных.
